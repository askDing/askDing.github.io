    web相关基础知识

    HTTP

    • http协议是 超文本传输协议

    • http协议的作用
      在浏览器和服务器间进行交互(访问、读取、文件交互等等、)

    HTTP数据包

    • 请求头 : 客户机的信息(请求地址、浏览器、操作系统等等)
    • 响应头 : 服务器返回的信息(搭建平台、脚本语言、文件类型等等)

    URL理解

    http://www.test.com/news/1.php?id=1

    http:// 协议
    www.test.com 域名
    /news/ 代表news目录
    1.php 脚本文件
    ? 脚本与参数之间的连接符
    Id 参数
    1 参数值

    http协议进去到www.test.com/下面的/news/这个目录下的1.php文件，文件里的值为1的id参数。

    HTTP状态码

    • 2xx
      • 200 访问正常
    • 3xx
      • 302 重定向，(访问A可能会跳转到B)
    • 4xx
      • 403 代表目录存在，但是不允许访问内容
      • 404 代表不存在的访问请求(文件、目录)。
    • 5xx
      • 500 由于服务器内部错误，导致无法访问我们的请求

    • 服务器系统探测方法

      • IIS是windows服务器
        • IIS6.0 windows2003
        • IIS7 7.5 基本都是windows2008搭建
        • IIS8 8.5 windows2008以上版本(IIS8 一般都是win2012搭建)
      • Apache和nginx 可能windows/linux (nginx 和linux 都可以是80 8080默认端口)

    • 利用linux特性来判断

      • Linux特性:它对大小写敏感
        如果linux服务器的站 www.test.com/abc.php 访问www.test.com/abC.php 就会报错。
        因为服务器找不到这个文件，对大小写敏感。
        Windows:对大小写不敏感。

      • 如果访问网站存在的一个文件，文件名改成大写，如果访问正常就说明是windows系统，反之是linux系统。

    • Webshell密码查找
      1. 搜索font 在高亮偏后的位置，可能会出现密码。
      2. 输入一个错误的密码。看源码font

    原理:由于很多，马会有后门。一部分在url get中传递出去，这样给他profile=a 可以溢出密码(必须是马内的参数)

    • URL中关于空格的编码正是与空格所在位置相关
      • 空格被编码成加号+的情况只会在查询字符串部分出现
      • 空格编码成%20则可以出现在路径和查询字符串中

    原因：
    urlencode(“ “) 返回+
    encodeURI(“ “) 返回%20

    • W3C标准规定：
      当 Content-Type：application/x-www-form-urlencoded时，URL中？后出现的空格用+替代（即URL中查询参数名和参数值出现的空格）
    • RFC 2396规定：
      URI里的保留字符都需转义成%HH格式，
      空格会被编码成%20，
      加号+本身也作为保留字而被编成%2B，

    所以一个安全的举措是URL中统一使用%20来编码空格字符。
